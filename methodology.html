GEO Workbench Lab Methodology

This page explains the methodology used by GEO Workbench Lab to test page-quality readiness and AI answer evidence.

1. What the methodology is

The GEO Workbench Lab methodology is a repeatable before/after workflow:

  1. select or publish a target page;
  2. crawl and extract the page;
  3. audit its page-quality and technical signals;
  4. collect baseline AI answers with fixed prompts;
  5. map prompt failures to page gaps;
  6. create a claim-safe optimization patch;
  7. publish only after review;
  8. run after crawl, extraction, and audit;
  9. retest the same prompts;
  10. report what improved and what did not.

2. Why baseline evidence is required

Without baseline evidence, a page update can look better but still fail to change AI answers. Baseline evidence records what models said before the update, which tools they recommended, which URLs they cited, and whether the target page appeared naturally.

Baseline evidence should include exact prompt, platform, model name, timestamp, raw answer text, screenshots, cited URLs, target URL accuracy, and competitor mentions.

3. Before crawl, extraction, and audit

The before phase checks whether the target page is technically reachable and extractable:

  • HTTP status;
  • final URL;
  • robots.txt access;
  • noindex state;
  • canonical URL;
  • sitemap availability;
  • title and H1;
  • visible FAQ;
  • tables and structured content;
  • schema JSON-LD;
  • llms.txt.

In this pilot, the weak before page was public and crawlable but intentionally lacked several GEO-readiness blocks.

4. AI answer ledgers

AI answer ledgers store one row per prompt/platform observation. A completed row should include:

  • prompt ID;
  • exact prompt;
  • platform;
  • model name;
  • timestamp;
  • raw answer text path;
  • screenshot path;
  • whether GEO Workbench Lab was mentioned;
  • whether it was recommended;
  • whether the target URL was cited;
  • cited URLs;
  • competitors mentioned;
  • recommendation strength;
  • evidence quality score.

Rows that are blocked, unavailable, or incomplete should remain in the ledger rather than being deleted.

5. Prompt-to-page gap mapping

A prompt-to-page gap is the difference between what a user asks and what the target page clearly supports. For example:

Prompt typeFailure modePossible page gap
discoverybrand not surfacedweak entity/category signal
recommendationknown tools dominateunclear use-case fit or weak authority
trustprivacy unclearmissing data-boundary explanation
comparisonenterprise tools dominatemissing comparison context
citationtarget URL not citedweak source block or insufficient external discovery

6. Claim-safe patching

A claim-safe patch adds page sections only when they map to evidence. It avoids fake users, fake awards, fake security claims, fake reviews, and unsupported performance claims.

Allowed claims include the self-owned pilot status, workflow description, observed page-quality metrics, and observed no-lift result. Stronger claims require evidence.

7. After crawl and audit

After publication, the same crawl/extraction/audit checks run again. In this pilot, the page-quality audit score improved from 60 to 95.

The after page added or improved:

  • schema JSON-LD;
  • llms.txt;
  • visible FAQ;
  • comparison tables;
  • citation-friendly source block;
  • limitations;
  • methodology language;
  • internal source structure.

8. After AI retesting

After AI retesting uses the same prompts as baseline. The goal is not to force success, but to measure whether mention rate, recommendation rate, citation rate, target URL accuracy, competitor dominance, and recommendation strength changed.

In this pilot, 12 ChatGPT after-test answers were collected manually. Screenshots were missing and the model name was unknown.

9. Result classifications

ClassificationMeaning
PAGE_QUALITY_IMPROVEDCrawl/extraction/audit or content-quality signals improved.
AI_LIFT_OBSERVEDAI answer metrics improved with comparable before/after evidence.
AI_LIFT_NOT_OBSERVEDAI answer metrics did not improve.
INCONCLUSIVEEvidence quality is too weak, missing, or inconsistent.

10. What this pilot proved

This pilot proved that a weak public page can be turned into a stronger source-like page with measurable page-quality improvement.

Observed page-quality change:

  • audit score: 60 → 95
  • schema JSON-LD: 0 → 1
  • llms.txt: 404 → 200
  • FAQ count: 3 → 7
  • table count: 0 → 2

11. What this pilot did not prove

This pilot did not prove immediate ChatGPT AI lift. The 12-prompt after-test did not show improved natural mention, recommendation, citation, or target URL accuracy.

It also did not prove Perplexity, Gemini, Claude, or Google AI Overview lift.

12. Why GEO optimization is iterative

AI answer systems use changing retrieval, ranking, source, and model behavior. A page can be technically strong and still fail natural discovery if the entity has weak external evidence. That is why the next iteration focuses on supporting source assets, indexing signals, and entity evidence rather than another homepage rewrite.
